# Архитектурный принцип: zero-build React + PHP

**Короткая формулировка:** весь фронтенд — это React, работающий напрямую в браузере без билда и без Node.js на сервере. Бэкенд — WordPress/PHP на обычном shared hosting. Всё живёт на одном домене. Никакой отдельной инфраструктуры, никакого VPS, никакой мультидоменной архитектуры.

Этот документ объясняет, **почему** мы пришли именно к такой архитектуре, какие проблемы она решает, и где у неё границы. Это не просто технический приём — это сдвиг в том, как думать о стеке в эпоху AI-ассистентов.

---

## 1. Проблема

Типичная задача современного небольшого бизнеса:

- Нужен красивый, современный, функциональный сайт — лендинг, продуктовая страница, клиентская админка, форма заявок.
- UI должен быть уровня 2020-х: анимации, реактивность, интерактивные компоненты, состояние, тёмная тема — короче, то, что **ожидают** посетители в 2026 году.
- Есть **shared hosting** (типично Hostinger, SiteGround, GoDaddy, Bluehost — десятки аналогов) стоимостью ~$3–10/мес с установленным WordPress.
- Бюджет и объём задач **не оправдывают** содержание VPS, отдельного Node-сервера или платформы типа Vercel/Netlify.
- Хочется **один домен**, один TLS-сертификат, один деплой, один кабинет — без зоопарка сервисов.

Казалось бы — противоречие. «Современный UI» → требует React → требует Node → требует VPS → противоречит «один простой shared hosting за $5». Отсюда распространённое заблуждение: «у нас маленький бизнес, красивый сайт нам не по карману».

## 2. Как это типично решают

Индустриальная практика последних 5–7 лет предлагает несколько шаблонных путей. Каждый имеет известные минусы:

### 2.1. Остаться на чистом WordPress + PHP

Шаблоны тем, плагины (Elementor, WPBakery, Divi), классический серверный рендеринг. Это самое простое, но UI получается **ограниченным** — он ощущается как «сайт из 2015». Сложная интерактивность, плавные переходы, кастомные компоненты — либо невозможны, либо требуют мучительной ручной работы с jQuery-эпохи.

Это путь большинства малого бизнеса. Отсюда и визуальная «усталость» десятков миллионов WP-сайтов — все выглядят одинаково плоско.

### 2.2. SPA на React + Node.js-бэкенд

«Правильный современный подход», который описан в 90% туториалов 2019–2024 годов. React-приложение на фронте + Express/Fastify/NestJS на бэке + MongoDB/PostgreSQL + всё это в Docker-контейнерах.

**Конфликт:** shared hosting на такое не рассчитан. Нужен VPS (~$10–50/мес минимум, плюс обслуживание), процесс-менеджер (PM2), реверс-прокси (nginx), мониторинг, бэкапы, SSL через Let's Encrypt, обновления ОС. Для **одного лендинга** — это из пушки по воробьям.

### 2.3. Next.js + Vercel/Netlify

Серверный рендер React на managed-платформе. Чисто, современно, хорошее SEO. Бесплатно на малых объёмах, но:

- Бэкенд ты теперь **не контролируешь** — Vercel даёт serverless-функции с ограничениями.
- Если у тебя уже **есть WordPress** и ты его не собираешься выкидывать — получаешь два сервера (WP на Hostinger + React на Vercel), два домена или сложную проксификацию, CORS-конфликты, и вместо одного деплоя — два.
- Нужно выучить отдельную платформу, её соглашения, её ограничения (cold starts, лимиты функций, биллинг).

### 2.4. Мультидоменная архитектура — VPS для Node + shared hosting для WP

Классическая «страдальческая» конфигурация, про которую в том числе думал владелец этого проекта до сегодняшнего дня:

- `yoursite.com` → WordPress (shared hosting)
- `app.yoursite.com` или отдельный домен → React SPA на VPS с Node
- Между ними CORS, общий JWT, синхронизация пользователей, две админки, два деплоя, два мониторинга.

Каждый компонент тянет свою инфраструктуру. Каждый интеграционный шов — точка отказа. **Головная боль, которая поглощает силы, которые должны уходить в продукт.**

### 2.5. Headless WordPress + Gatsby/Next.js

WP как data API + статический генератор React → деплой на CDN. Технически красиво, но:

- Рабочий цикл усложняется: правка в WP → rebuild статики → деплой. Мгновенность теряется.
- Опять нужен Node (локально для билда или на хостинге).
- Оправдано для больших контент-проектов, для лендинга — избыточно.

## 3. Конфликт: Node.js и shared hosting

Корень всех проблем выше — **фундаментальная несовместимость** Node.js с архитектурой shared hosting:

| Shared hosting (Apache/LiteSpeed + PHP) | Node.js |
|---|---|
| Каждый запрос порождает короткоживущий процесс | Долгоживущий серверный процесс |
| PHP-файлы = статические скрипты, Apache их читает | Нужен процесс-менеджер (PM2, systemd) |
| Никакого open-порта, всё через Apache на 80/443 | Нужно слушать порт + реверс-прокси |
| Готовый MySQL + cPanel + email + cron | Всё надо поднимать руками |
| $3–10/мес | VPS от $5 за минималку, реально $15–50+ |
| Автоматический SSL, автообновления | Ответственность на тебе |

На shared hosting **физически невозможно** запустить Node — он требует управления процессами, которых ты не контролируешь. Хостинги, которые поддерживают Node (Hostinger Cloud, A2 Hosting), стоят в 3–5 раз дороже и всё равно это не настоящий shared.

Отсюда **дилемма**, в которой десятилетие сидели тысячи владельцев WordPress-сайтов:

> «Либо я остаюсь с уродливым PHP-фронтендом, либо мигрирую на отдельный VPS ради одного React-компонента.»

## 4. Наш подход: React БЕЗ билда + PHP

Решение, которое **снимает конфликт целиком**.

### Суть

- **Frontend:** React, но без сборочного пайплайна. React, ReactDOM, Babel Standalone загружаются с CDN (unpkg.com) прямо в браузер. JSX-файлы компилируются **на клиенте** при загрузке страницы. Результат — те же React-компоненты, те же hooks (`useState`, `useEffect`), та же мощь экосистемы, но **конечный продукт — это просто HTML-файлы с ссылкой на .jsx**. Статические ресурсы, которые без изменений лежат на любом веб-сервере.

- **Backend:** WordPress + PHP. WordPress — это, по сути, готовый бэкенд-фреймворк с базой данных, системой ролей, REST API (`/wp-json/`), email-рассылкой, плагинами на все случаи жизни. Он **уже есть** на shared hosting. Его не надо ставить, настраивать, поддерживать — всё это делает Hostinger.

- **Связь:** React-фронт вызывает `fetch('/wp-json/...')` или кастомные PHP-эндпоинты. WordPress обрабатывает и возвращает JSON.

- **Размещение:** всё на **одном домене**, в одном `public_html/`. React-файлы в корне и в папках, WP в своих стандартных директориях. Apache через `DirectoryIndex index.html index.php` решает, кого отдавать первым.

### Как это выглядит на этом проекте

```
public_html/                       (один корень сайта)
├── index.html                     ← React-лендинг (статика!)
├── preview/index.html             ← другая React-страница
├── landing/
│   ├── data.js                    ← данные для React
│   └── components.jsx             ← React-компоненты (16 штук)
├── wp-config.php                  ← WordPress-бэкенд
├── wp-admin/                      ← админка WP
├── wp-content/
├── wp-json/                       ← автоматический REST API
└── .htaccess                      ← один файл правит маршрутизацией
```

**Один домен. Один деплой. Одно место.** `scp` файлов через SSH — весь DevOps.

### Работа разработчика

```
Локально:
  python3 -m http.server 4000       ← простой HTTP-сервер для локального превью
  [редактируешь .jsx / .html]
  [F5 в браузере]                   ← видишь изменения мгновенно, без build
  git commit && git push            ← версионируешь

Деплой:
  scp file user@host:~/public_html/ ← заливаешь изменённый файл
  wp litespeed-purge all            ← сбрасываешь кэш LiteSpeed
  [готово]
```

Никакого `npm install`, никакого `npm run build`, никаких `node_modules`, никаких Docker-контейнеров, никаких CI/CD-пайплайнов. Одно действие — одно изменение.

## 5. Как это решает перечисленные проблемы

| Проблема | Решение |
|---|---|
| Современный UI требует React | React тут есть, полноценный, со всеми hooks и с любой сложностью |
| React требует Node и билд | Не требует — Babel компилирует JSX на клиенте |
| Node не работает на shared hosting | Node **не нужен нигде** — ни локально, ни на сервере |
| Нужна мультидоменная инфра | Один домен, одно `public_html/` |
| Нужен отдельный VPS | Не нужен — shared hosting закрывает всё |
| CORS между фронтом и бэком | Их нет — один домен |
| Дорогой хостинг | $3–10/мес |
| Сложный DevOps | `scp` + `wp-cli` + `.htaccess`. Всё. |
| Надо учить Next.js, Vercel, Docker | Не надо. HTML, React, PHP. |

## 6. Плюсы

1. **Радикальная простота операций.** Нет Node, нет build-pipeline, нет CI. Файл → `scp` → работает.
2. **Мгновенный dev-цикл.** Правишь `.jsx` → перезагрузил страницу → видишь результат. Никакого `npm run dev`, никакого ожидания HMR.
3. **Полный функционал React.** Hooks, context, кастомные компоненты, сторонние библиотеки (если есть CDN-версия). Никаких компромиссов по возможностям UI.
4. **WordPress из коробки.** Авторизация, пользователи, роли, email, формы, SEO, медиа-библиотека — всё это **уже есть**, плагины на всё.
5. **Один домен, один TLS, одна админка хостинга.** Минимум когнитивной нагрузки.
6. **Нет CORS.** Принципиально упрощает жизнь.
7. **Дёшево.** $3–10/мес за хостинг. И за этот бюджет ты получаешь полноценный современный веб-стек.
8. **Совместим с AI-ассистентами.** AI может писать и править любой файл проекта без необходимости понимать build-систему, dependency-граф, webpack-конфиги. Проект **плоский и читаемый** целиком.
9. **Версионирование простое.** Git хранит всё, что есть на сайте. Деплой — это буквально копия файлов на сервер.
10. **Портабельность.** Этот сайт можно перенести на любой другой хостинг за 10 минут — просто скопировать файлы.

## 7. Минусы и ограничения

Быть честным: у подхода есть реальные границы, и важно их понимать.

### 7.1. Babel-в-браузере — нагрузка на клиента

Каждый раз при загрузке страницы Babel Standalone компилирует все `.jsx`-файлы прямо в браузере. Для нашего проекта (16 компонентов, ~800 строк) это ~100–200 мс на современном устройстве, на мобильном может быть 300–500 мс. Заметно, но не катастрофично.

**Когда становится проблемой:** когда компонентов становится 50+, кодовая база > 5000 строк. Тогда задержка растёт до секунд — пользователь видит белый экран.

**Что делать:** переходить на **build локально**. Ставишь Vite на свою машину, собираешь React в обычный JS-бандл (минифицированный, без Babel), заливаешь уже `dist/` на shared hosting. **Node появляется только на твоём ноутбуке, на сервере по-прежнему ничего не меняется.**

### 7.2. Зависимость от CDN при загрузке

React, Babel, Tailwind тянутся с unpkg.com / cdnjs при каждой загрузке. Типично ~500 KB. Браузер кэширует, но первый визит — это трафик.

**Смягчает:** CDN обычно быстрые, Tailwind можно заменить на prebuilt-CSS, можно хостить копии у себя.

**Что делать в крайнем случае:** скачать React и Babel локально, положить в `/vendor/`, грузить оттуда. Тогда вообще ни на какие внешние сервисы не завязан.

### 7.3. npm-экосистема ограничена CDN-доступными пакетами

Не все npm-пакеты выложены в CDN-формате (UMD или ESM.sh). Если тебе нужен специфичный пакет без CDN-варианта — либо обёртывать его, либо переходить на build.

**В реальности:** все массовые библиотеки (React Router, Framer Motion, date-fns, lodash, chart.js, three.js и т.д.) имеют CDN-версии. Редкая узкоспециализированная — может не иметь.

### 7.4. TypeScript без build недоступен

Строгая типизация требует компилятора `tsc`, а он требует Node. Если хочешь TypeScript — нужен build-шаг.

**Смягчает:** для небольшого проекта JS + JSDoc-комментарии закрывают большую часть потребности в типизации. AI-ассистенты компенсируют отсутствие типов (обнаруживают логические ошибки, предлагают исправления).

### 7.5. SEO и время первой отрисовки (FCP)

React рендерит в браузере — до выполнения JS пользователь видит пустой HTML. Поисковые краулеры Google в целом умеют выполнять JS, но это **медленнее и менее надёжно**, чем готовый HTML.

**Когда это критично:** контентные сайты с сотнями страниц, блоги, магазины.
**Когда не критично:** лендинг, бизнес-карточка, админка (её не индексируют).

**Что делать, если критично:** либо переходить на **Astro/Next.js со static export** (они генерят готовый HTML на стадии билда, Node снова нужен только локально), либо использовать prerender.io / похожие сервисы, либо оставить для контентных страниц обычный WordPress-рендеринг (это вообще один из паттернов — смешивать: простые страницы через WP, сложная интерактивность через React).

### 7.6. Нет hot reload с сохранением состояния

Правишь компонент → F5. Состояние компонента сбрасывается. В современных build-системах (Vite) есть HMR, который обновляет компонент без потери состояния. Без build — нет.

**В реальности:** для компонентов с небольшим состоянием раздражение минимальное. Для сложного дебага формы — лучше вынести состояние в `localStorage` на время.

### 7.7. Минификация/оптимизация

Без билда не происходит автоматической минификации, tree-shaking, code-splitting. Ты отдаёшь исходный `.jsx` (который сам по себе небольшой) плюс Babel компилирует в JS тяжелее, чем оптимизированный бандл.

**В реальности:** для проекта < 10 KB исходников это не имеет значения. Для 100 KB+ — начинает.

## 8. Что теперь НЕ актуально (пост-AI эпоха)

AI-ассистенты (Claude, Copilot, Cursor) снимают целый пласт архитектурных ограничений, которые формировали «здравый смысл» индустрии предыдущие 10 лет. Список вещей, которые **перестали быть аргументами**:

### 8.1. «У команды один язык — JavaScript»

До AI: переключение с JS на PHP требовало переобучения, потери скорости, багов от незнания идиом.
После AI: AI пишет одинаково хорошо на TS, PHP, Python, Go. Барьер исчез.

### 8.2. «Мы не знаем PHP / WordPress API»

Раньше: приходилось читать документацию, набираться опыта, учиться паттернам Laravel/WP. Недели или месяцы.
Теперь: AI знает эти экосистемы лучше среднего разработчика. Спрашиваешь — получаешь рабочий код. Обучение происходит **в процессе использования**, а не до.

### 8.3. «React = Next.js = Vercel = современный стек»

Это был **бандл решений**, навязанный хайпом. В эпоху до AI логика была: «все используют Next.js → про него много туториалов → легче нанимать → значит, это стандарт». После AI: стек не нужно выбирать по распространённости, потому что AI помогает в любом.

### 8.4. «Нужно выучить Docker и Kubernetes»

Раньше: без знания контейнеров ты не поднимешь современное приложение. Теперь: если задача закрывается shared hosting — контейнеры просто не нужны, и AI это видит.

### 8.5. «Нужен DevOps-инженер»

Раньше: CI/CD, IaC, мониторинг — отдельная специализация. После AI + простая архитектура: deploy = `scp`, мониторинг = uptime robot. Один человек справляется.

### 8.6. Вкусовщина и хайп

«Модно», «современно», «передний край» — эти категории в эпоху AI **не несут смысла** при выборе стека. AI равнодушен к моде. Решение принимается по объективным признакам: **задача + инфраструктура + границы возможностей**.

## 9. Когда этот подход ПРАВИЛЬНО подходит

- **Размер команды:** 1 человек + AI. Маленькие команды (2–5 человек) тоже, но при росте — нужен build.
- **Инфраструктура:** уже есть shared hosting с WordPress (или легко доступен).
- **Задачи:** лендинги, корпоративные сайты, формы заявок, админки CRUD, пользовательские кабинеты, простые дашборды, клиентские порталы, контент-сайты с умеренной интерактивностью.
- **Критичность SEO:** средняя или низкая (для критического SEO — Astro SSG / Next.js export сверху той же идеи).
- **Бюджет:** оптимизированный. Когда каждый лишний сервис стоит больно.
- **Приоритет:** быстро запустить, быстро итерировать, минимум инфраструктурной возни.

## 10. Когда НЕ подходит

- **Real-time** (чаты, живая совместная работа, коллаборативные редакторы) — PHP плох с долгоживущими соединениями. Нужен Node / Go / Elixir + WebSocket.
- **Тяжёлые ML/AI-вычисления на бэке** — Python на выделенном сервере с GPU. Shared hosting тут бесполезен.
- **Высоконагруженные сервисы** — миллионы RPS, миллисекундные SLA. Shared hosting уйдёт в отказ.
- **Строгое SSR для SEO-чувствительного контента** — магазин на 50 тысяч SKU. Нужен полноценный Next.js / нативный WP с правильной темой.
- **Команда уже живёт в Next.js/Node-экосистеме** — нет смысла ломать сложившийся процесс ради экономии $10/мес.
- **Корпоративные требования** — compliance, SOC2, изоляция — обычно требуют enterprise-инфраструктуры.

## 11. Стратегия масштабирования

Этот подход **не заставляет** выбирать между «простотой сейчас» и «гибкостью потом». У него чёткий путь роста:

1. **Старт:** zero-build React + WP (текущее состояние).
2. **Рост (> 50 компонентов):** добавляешь Vite локально для сборки. Результат (`dist/`) заливается на тот же shared hosting. **На сервере ничего не меняется.** Node нужен только на твоём ноутбуке.
3. **SEO становится критичен:** переходишь на Astro или Next.js в режиме static export. Результат — опять статические HTML на shared hosting. WP продолжает быть бэкендом.
4. **Нужен real-time или тяжёлый бэк:** добавляешь **один** специализированный сервис (Node/Python на VPS, Supabase, Cloudflare Workers), не переписывая всё остальное. Остальная инфра как была.
5. **Только в крайнем случае:** полная миграция на Vercel/AWS/Kubernetes. Но к этому моменту у тебя уже бизнес такого масштаба, что это оправдано.

На каждом шаге ты **продлеваешь** жизнь предыдущей архитектуры, а не выбрасываешь её.

## 12. Индустриальные параллели

Этот подход — не авторское изобретение. Он ложится на существующие паттерны:

- **«Headless WordPress»** — WP как бэкенд, любой фронт сверху. Практика с 2015 года. Документирована в WP-сообществе.
- **«JAMstack»** — JavaScript + APIs + Markup. Классификация сайтов, где динамика достигается через API-вызовы со статического фронта. Термин введён около 2016 года.
- **«Progressive enhancement»** — страница работает без JS, React накладывается сверху. Принцип, который старше React.
- **«Islands architecture»** — идеология Astro: статический HTML + вкрапления интерактивных компонентов. Формализация того, что мы делаем.
- **«React без инструментов»** — официальная документация React в 2024 году добавила раздел про использование React прямо из CDN без билда.
- **Стек LAMP с React-фронтом** — Laravel/Symfony/CakePHP + React — стандартная связка в PHP-мире уже лет 7.

Что изменилось сейчас — **AI делает этот подход доступным для одиночки без глубокого бэкграунда**, а раньше требовалась инженерная зрелость в нескольких стеках одновременно.

## 13. Практический вывод для нашего проекта и будущих

Когда стоит задача сделать **ещё один сайт** (лендинг, интернет-магазин, клиентская админка, учебная платформа, портфолио, блог, etc.), и встаёт вопрос «а на чём это всё поднимать» — **по умолчанию** ответ:

1. Тот же домен или поддомен на том же shared hosting.
2. Новая папка в `public_html/` с React-компонентами.
3. WordPress (или чистый PHP) для серверной логики — формы, email, база.
4. Если сайт независим от WP — даже проще: только React-файлы + пара PHP-эндпоинтов.

**Не нужно** заранее проектировать «мультидоменную инфраструктуру», «выделенный Node-сервер для такого-то сервиса», «интеграцию через API между доменами». Всё это решается **внутри одного shared hosting**, и масштабируется, когда понадобится.

Это **освобождает** голову для работы над продуктом, а не над инфраструктурой.
